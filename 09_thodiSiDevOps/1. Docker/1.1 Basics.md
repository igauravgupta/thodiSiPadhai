## Hypervisor

- A hypervisor is a software layer that enables virtualization, allowing multiple virtual machines (VMs) to run on a single physical host by abstracting and managing hardware resources. There are two types:
  - **Type 1 (bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V).
  - **Type 2 (hosted):** Runs on top of an operating system (e.g., VirtualBox, VMware Workstation).

### Relation to Docker and Containerization

- Docker uses containerization, not full virtualization. Instead of running full OS instances like hypervisors do with VMs, Docker containers share the host OS kernel, making them more lightweight and faster to start.
- Hypervisors virtualize hardware → run VMs with their own OS.
- Docker virtualizes the application environment → run containers that share the host OS.
- **Note:** VMs via hypervisors are heavier, isolated at the hardware level; containers via Docker are lightweight, isolated at the OS level.

## Docker Engine

- The Docker Engine is the core client-server technology that enables containerization. It allows developers to build, run, and manage containers on a host system. Containers share the host OS kernel but are isolated in terms of processes, file system, and networking.
- Docker Engine uses OS-level virtualization.

## Running a container on Docker Engine

1. **Enter the image pull command in the terminal:**  
   `docker pull hello-world`
2. **Docker Daemon checks if the image exists locally.**  
   If not, it pulls the image from Docker Hub (or any configured registry).
3. **Docker uses the image as a template to create a container layer.**  
   This layer is writable, while the image remains read-only (copy-on-write model).
4. **Container Creation:** Docker allocates:
   - Filesystem (via UnionFS)
   - Namespaces (for isolation: PID, network, mount, etc.)
   - Cgroups (for resource limits: CPU, memory)
   - Networking (virtual bridge, IP assignment)
   - It sets up the container with the entrypoint defined in the image.
5. **Container Start:**  
    `docker run hello-world`
   - The init process inside the container starts. The container runs isolated but shares the host kernel.
6. **Lifecycle Management:**  
   Docker monitors the container via the daemon. You can manage it with commands like `docker stop`, `docker exec`, `docker logs`, etc.

## Basic Docker Commands

- `docker run<image_name>`: pull and run the Docker image
- `docker ps`: gives all the running containers
- `docker ps -a`: gives all the containers (running and stopped)
- `docker stop <container_id>`: stop running docker container
- `docker remove <container_id>`: remove container
- `docker images`: show all the images
- `docker rmi <image_id>`: remove docker image
- `docker run -d <image_name>`: pull and run the Docker image as a background process.

### Key Concepts in Docker

#### 📦 Docker Images

- “A Docker Image is a lightweight, standalone, and executable package that includes everything needed to run a piece of software: code, runtime, libraries, environment variables, and config files.”
- **Layered:** Built in layers, where each instruction in a Dockerfile forms a new layer. This optimizes storage and speeds up builds.
- **Immutable:** Once built, images do not change, ensuring consistency across all environments (dev, test, prod).
- **Portable:** Can run anywhere — on any system with Docker installed, ensuring platform independence.

#### 🧱 Docker Containers

- A Container is a running instance of an image.
- Containers are isolated environments that share the host’s kernel but have separate namespaces and cgroups.
- They start from the image’s filesystem and add a writable layer on top.
- **Interview Phrase:** “A container is a live, running version of a Docker image, isolated and reproducible.”

#### 📝 Dockerfile

- A Dockerfile is the build script used to create a Docker image.
- It defines a base image, instructions like copying files, installing dependencies, and setting entry points.
- **File name:** Dockerfile
- **Build Docker image:**  
   `docker build -t <image_name> <directory_where_dockerfile_present>`
- **Run Docker Image:**  
   `docker run -p <host_port_no>:<conatiner_port_no> <image_name or image_id>`

### Scenario: Collaborative Development with Docker

- You and your teammate are working on a Node.js project. You’ve containerized the app using a Dockerfile and pushed the whole project to GitHub. Your teammate clones the repo, opens the code in their local IDE (like VS Code), and runs the app using Docker.
- For live development, they use a bind mount to sync local code with the container:  
   `docker run -v $(pwd):/app -p 3000:3000 node-app-image`
- This lets them see code changes instantly in the running container, without rebuilding every time. This setup ensures consistent environments while keeping local development fast and flexible.

- ✅ **Best practice:** Code outside the container, run inside Docker.

- For special cases (like remote dev or pre-configured tools), they may enter the container interactively using:  
   `docker run -it node-app-image /bin/bash`
- But this approach isn’t preferred for regular dev because changes are lost unless committed and synced back.
