### 1. What is Docker and why do we use it?

Docker is an open-source platform for automating the deployment, scaling, and management of applications using containerization. Containers package code and dependencies together, ensuring consistency across environments. Docker is used for portability, isolation, efficient resource usage, and simplified deployment.

### 2. Difference between a container and a virtual machine

Containers share the host OS kernel and isolate applications at the process level, making them lightweight and fast. Virtual machines emulate hardware and run separate OS instances, which are heavier and slower to start.

### 3. Image vs. Container

An image is a read-only template with instructions for creating a container. A container is a running instance of an image, with its own filesystem, processes, and network.

### 4. Creating a Docker container from a Node.js app

Write a Dockerfile specifying the base image (e.g., node:18), copy app files, install dependencies, and set the start command. Build the image and run a container using `docker build` and `docker run`.

### 5. Command to build a Docker image

Use `docker build -t <image-name> .` to build an image from a Dockerfile in the current directory.

### 6. Running a container on a specific port

Use `docker run -p <host-port>:<container-port> <image-name>` to map container ports to host ports.

### 7. CMD vs ENTRYPOINT in Dockerfile

CMD sets default arguments for the container, which can be overridden. ENTRYPOINT sets the main command to run, and arguments are appended. ENTRYPOINT is preferred for fixed commands.

### 8. Listing all running Docker containers

Use `docker ps` to list running containers.

### 9. docker ps -a command

`docker ps -a` lists all containers, including stopped ones.

### 10. docker stop vs docker kill

`docker stop` gracefully stops a container, allowing cleanup. `docker kill` forcefully terminates it immediately.

### 11. Dockerfile and its key instructions

A Dockerfile is a script with instructions to build an image. Key instructions: FROM, WORKDIR, COPY, RUN, CMD, ENTRYPOINT, EXPOSE, ENV.

### 12. Explaining each line of a Dockerfile

Each line sets up the environment, copies files, installs dependencies, exposes ports, and defines the start command.

### 13. Use of WORKDIR

WORKDIR sets the working directory for subsequent instructions, improving readability and maintainability.

### 14. COPY before RUN npm install

COPY copies files before RUN npm install to ensure dependencies are installed from the copied package.json.

### 15. RUN npm install vs RUN npm ci

`npm install` installs dependencies, possibly updating package-lock.json. `npm ci` installs exactly as specified in package-lock.json, ensuring reproducible builds.

### 16. Using --omit=dev in Dockerfile

`--omit=dev` skips installing dev dependencies, reducing image size and attack surface for production images.

### 17. Reducing Docker image size

Use slim base images, multi-stage builds, .dockerignore, and avoid unnecessary files and dependencies.

### 18. Multi-stage build in Docker

Multi-stage builds use multiple FROM statements to separate build and runtime environments, copying only necessary files to the final image.

### 19. Using multi-stage Dockerfile

Multi-stage Dockerfiles reduce image size and improve security by excluding build tools and unnecessary files from the final image.

### 20. Copying only dist/ to final image

Copying only the dist/ directory ensures only built artifacts are included, reducing size and exposure of source code.

### 21. EXPOSE in Dockerfile

EXPOSE documents which ports the container listens on, but does not publish them.

### 22. EXPOSE 8000 vs -p 3000:8000

EXPOSE 8000 is for documentation. `-p 3000:8000` maps host port 3000 to container port 8000, making it accessible externally.

### 23. Container communication

Containers communicate via networks, using internal DNS to resolve service names.

### 24. Docker network

A Docker network connects containers, allowing communication. Use it for multi-container apps and isolation.

### 25. Passing environment variables to containers

Use `-e VAR=value` with `docker run` or ENV in Dockerfile to set environment variables.

### 26. ENV in Dockerfile vs -e during run

ENV sets variables at build time, available in all containers. `-e` sets variables at runtime, overriding ENV if needed.

### 27. Managing secrets in Docker

Use Docker secrets (Swarm), environment variables, or external secret managers. Avoid hardcoding secrets in images.

### 28. Volume in Docker

A volume is persistent storage managed by Docker, used to store data outside the container lifecycle.

### 29. Creating and mounting a volume

Create with `docker volume create <name>`, mount with `docker run -v <name>:/path/in/container`.

### 30. Bind mount vs named volume

Bind mounts map host directories, named volumes are managed by Docker. Named volumes are preferred for portability.

### 31. Docker Compose

Docker Compose defines and runs multi-container applications using a YAML file.

### 32. docker-compose.yml for multi-container apps

docker-compose.yml specifies services, networks, and volumes, simplifying orchestration of multiple containers.

### 33. Explaining a Docker-Compose file

Each service defines its image, ports, environment, volumes, and dependencies. Networks and volumes are shared as needed.

### 34. Running multiple services with Compose

Use `docker-compose up` to start all defined services together.

### 35. Handling dependencies with Compose

Define dependencies using `depends_on` and service links, ensuring correct startup order.

### 36. Viewing Docker container logs

Use `docker logs <container-name>` to view logs.

### 37. Debugging a running container

Use `docker exec -it <container-name> sh` or `bash` to run commands inside the container.

### 38. Entering a running container

Use `docker exec -it <container-name> sh` or `bash` for an interactive shell.

### 39. Checking container listening ports

Use `docker port <container-name>` or inspect the container.

### 40. Docker best practices

Use minimal base images, multi-stage builds, .dockerignore, avoid root, keep images small, and automate builds.

### 41. Importance of .dockerignore

.dockerignore excludes files from the build context, speeding up builds and reducing image size.

### 42. Avoiding dev dependencies in production images

Dev dependencies increase size and risk. Only production dependencies should be installed.

### 43. Keeping image size minimal

Use slim images, multi-stage builds, .dockerignore, and remove unnecessary files.

### 44. Avoid running containers as root

Running as root increases security risks. Use non-root users for better isolation.

### 45. Using Docker in a project

Docker standardizes environments, simplifies deployment, and enables reproducible builds and testing.

### 46. Containerizing a MERN stack app

Containerization isolates each service (MongoDB, Express, React, Node), simplifies orchestration, and improves portability.

### 47. Docker for deployment and testing

Docker enables consistent environments for deployment and automated testing, reducing "works on my machine" issues.

### 48. Issues faced with Docker and solutions

Common issues: networking, permissions, image size. Solutions: use networks, non-root users, multi-stage builds, and .dockerignore.

### 49. Deploying Docker containers to cloud

Use services like AWS ECS, Heroku, Azure Container Instances, or GCP Cloud Run to deploy containers.

### 50. Performance issues with containers

Issues: resource limits, slow builds, networking. Troubleshoot with monitoring, optimizing images, and tuning resource allocations.
