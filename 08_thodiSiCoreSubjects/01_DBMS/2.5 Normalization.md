## 1. What is Normalization?

Normalization is a process in database design used to organize data to reduce redundancy and improve data integrity. It involves dividing a database into tables and defining relationships between them.

## 2. Why Normalize?

- **Eliminate data redundancy**
- **Ensure data consistency**
- **Improve data integrity**
- **Simplify data maintenance**
- **Optimize queries**

## 3. Types of Normalization (Normal Forms)

### 1. First Normal Form (1NF)

- Each table cell contains a single value (atomicity).
- Each record is unique.

### 2. Second Normal Form (2NF)

- **Prime Attributes**: Attributes that are part of any candidate key.
- **Non-Prime Attributes**: Attributes that are not part of any candidate key.
- **Partial Dependency**: Occurs when a non-prime attribute is dependent on part of a composite primary key, not the whole key.
- **No Partial Dependency**: In 2NF, all non-prime attributes must depend on the entire primary key, not just a part of it.

- Must be in 1NF.
- All non-key attributes are fully functionally dependent on the primary key (no partial dependency).

### 3. Third Normal Form (3NF)

- **Transitive Dependency**: A transitive dependency occurs when a non-prime attribute depends on another non-prime attribute, which in turn depends on the primary key. In other words, if A → B and B → C, then C is transitively dependent on A through B.

- Must be in 2NF.
- No transitive dependency (non-key attributes depend only on the primary key).

### 4. Boyce-Codd Normal Form (BCNF)

- Stricter version of 3NF.
- For every functional dependency, the left side is a super key.

### 5. Fourth Normal Form (4NF)

- Must be in BCNF.
- No multi-valued dependencies.

### 6. Fifth Normal Form (5NF)

- Must be in 4NF.
- Deals with cases where information can be reconstructed from smaller pieces of data (join dependency).

### 7. Sixth Normal Form (6NF)

- Used in temporal databases.
- Each fact is stored in its own table.

---

**Summary Table**

| Normal Form | Main Rule                                  |
| ----------- | ------------------------------------------ |
| 1NF         | Atomic values, unique records              |
| 2NF         | No partial dependency                      |
| 3NF         | No transitive dependency                   |
| BCNF        | Super key on left of every dependency      |
| 4NF         | No multi-valued dependency                 |
| 5NF         | No join dependency                         |
| 6NF         | Temporal data, each fact in separate table |

## Disadvantages of Normalization

- **Complex Queries**: Highly normalized databases often require complex JOIN operations, which can make queries harder to write and understand.
- **Performance Overhead**: Multiple tables and joins can slow down query performance, especially for large datasets.
- **Increased Design Time**: Designing a normalized schema takes more time and effort.
- **Difficult Data Retrieval**: Retrieving related data from multiple tables can be cumbersome.
- **Potential Over-Normalization**: Excessive normalization may lead to too many tables, complicating database management.
