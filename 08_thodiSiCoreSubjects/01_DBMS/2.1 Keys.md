## Super Key

A **Super Key** is any combination of columns in a table that can uniquely identify a row.

- All candidate keys and primary keys are super keys.
- Super keys may include extra columns not necessary for uniqueness.

**Example:**  
If `StudentID` is unique, then `{StudentID}`, `{StudentID, Name}`, `{StudentID, Email}` are all super keys.

**Analogy:**  
Think of super keys as all possible ways to uniquely identify a row, even if some ways use more columns than needed.

---

## Candidate Key

A **Candidate Key** is a minimal super key—no unnecessary columns.

- Every candidate key is a super key, but not every super key is a candidate key.
- There can be multiple candidate keys in a table.

**Example:**  
In a `Student` table, both `StudentID` and `Email` can be candidate keys if each is unique.

**Analogy:**  
Candidate keys are the “best” super keys—just enough columns to guarantee uniqueness, no extras.

---

## Primary Key

A **Primary Key** is a candidate key chosen to uniquely identify rows in a table.

- Only one primary key per table.
- Cannot be NULL or duplicated.

**Example:**  
If `StudentID` is chosen as the primary key, it uniquely identifies each student.

**Analogy:**  
The primary key is the “winner” among candidate keys—the main unique identifier for the table.

---

## Alternate Key

An **Alternate Key** is a candidate key that was not chosen as the primary key.

- Still unique, can be used to identify rows.

**Example:**  
If `StudentID` is the primary key, `Email` is an alternate key.

**Analogy:**  
Alternate keys are “backup” unique identifiers.

---

## Foreign Key

A **Foreign Key** is a column that creates a link between two tables, referring to the primary key of another table.

- Used to maintain referential integrity.

**Example:**  
In an `Enrollment` table, `StudentID` can be a foreign key referring to the `Student` table.

**Analogy:**  
Foreign keys are “bridges” connecting tables.

---

## Composite Key

A **Composite Key** is a primary key made up of two or more columns.

- Used when a single column is not enough to uniquely identify a row.

**Example:**  
In an `OrderDetails` table, `OrderID` and `ProductID` together can be a composite key.

**Analogy:**  
Composite keys are “team efforts” for uniqueness.

---

## Unique Key

A **Unique Key** ensures all values in a column are different.

- Can have one NULL value (in most databases).
- Used to enforce uniqueness apart from the primary key.

**Example:**  
`Email` in a `Student` table can be a unique key.

**Analogy:**  
Unique keys are “guards” against duplicate values.

---

## Notes

- A foreign key must reference a column (or set of columns) in another table that is either a primary key or has a unique constraint (such as an alternate key).
- This ensures that every value in the foreign key column matches a unique value in the referenced table, maintaining referential integrity.
- In practice, a foreign key can point to an alternate key, not just the primary key, as long as the referenced key is unique.

---

## Summary

Keys are essential for organizing and managing data in a database. They help in uniquely identifying records, maintaining relationships, and ensuring data integrity. Understanding each type of key and their differences is important for interviews and practical database design.
