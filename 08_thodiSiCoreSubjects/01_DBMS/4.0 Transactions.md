## Introduction

A **transaction** is a sequence of one or more operations (such as read, write, update, or delete) performed on a database as a single logical unit of work. Transactions ensure data integrity by making sure that either all operations are completed successfully, or none are applied if an error occurs.

---

## Transaction Example

Consider a money transfer between two bank accounts. The transaction involves several steps:

- Reading the sender's account balance.
- Deducting the amount from the sender’s account.
- Adding the amount to the recipient’s account.

All these steps must be completed together. If any step fails, the database should revert to its previous state, preventing partial updates.

---

## Operations of a Transaction

A transaction consists of several types of operations that interact with the database:

### Read(X)

- **Definition:** Reads the value of a database element `X` and stores it in a temporary buffer for further processing.
- **Example:** When a user checks their account balance, a `Read` operation retrieves the balance from the database.

### Write(X)

- **Definition:** Writes or updates the value of database element `X` from the temporary buffer back to the database.
- **Example:** After a withdrawal, the updated balance is written back to the user's account in the database.

### Commit

- **Definition:** Finalizes a transaction, making all changes permanent in the database. Once committed, the transaction cannot be undone.
- **Example:** After a successful money transfer, a `Commit` operation ensures that the changes to both accounts are saved.

  ```sql
  COMMIT;
  ```

### Rollback

- **Definition:** Undoes all changes made by the current transaction, restoring the database to its last consistent state if an error occurs.
- **Example:** If there are insufficient funds during a transfer, a `Rollback` operation cancels all changes made during the transaction.

## ACID Properties in DBMS

![alt text](../../0.%20assests/DBMS-8.png)

The ACID properties ensure reliable processing of database transactions:

1. **Atomicity**

   - A transaction is "all-or-nothing": either all operations succeed, or none are applied.
   - If any part fails, the entire transaction is rolled back to maintain consistency.
   - **Commit:** If successful, changes are permanently applied.
   - **Abort/Rollback:** If failed, all changes are discarded.

2. **Consistency**

   - The database must remain in a valid state before and after a transaction.
   - All rules, constraints, and relationships (e.g., primary keys, foreign keys) must be preserved.
   - If a transaction violates any rule, it is rolled back to prevent invalid data.

3. **Isolation**

   - Transactions run independently, so changes made by one are not visible to others until committed.
   - Ensures the result of concurrent transactions is the same as if run sequentially.
   - Prevents issues like:
     - **Dirty reads:** Reading uncommitted data.
     - **Non-repeatable reads:** Data changes between two reads.
     - **Phantom reads:** New rows appear during a transaction.

4. **Durability**
   - Once committed, changes are permanently saved, even if the system fails.
   - Data is stored in non-volatile memory, allowing recovery to the last committed state.

## Transaction States in DBMS

![alt text](../../0.%20assests/DBMS-9.png)

A transaction in a DBMS goes through several states during its lifecycle:

1. **Active:** The transaction is currently executing its operations (read, write, etc.).
2. **Partially Committed:** The transaction has completed its final operation but changes are not yet permanent.
3. **Committed:** All changes made by the transaction are permanently saved in the database.
4. **Failed:** The transaction cannot proceed due to an error or violation of constraints.
5. **Aborted:** The transaction has been rolled back, and all changes are undone.

**State Transitions:**

- After execution, a transaction moves from _Active_ to _Partially Committed_.
- If all checks pass, it transitions to _Committed_.
- If an error occurs, it moves to _Failed_ and then to _Aborted_, undoing all changes.

These states help ensure the ACID properties and maintain database integrity.
