## Introduction

DBMS architecture defines the structure and organization of a database system. It determines how users interact with the database and how data is stored, processed, and retrieved.

---

## 1-Level Architecture

- **Single-tier architecture**
- The user interacts directly with the database.
- No abstraction between the user and the data.
- Suitable for small, simple applications.

### Deep Example

**Scenario:**  
A student uses a local SQLite database to store notes on their laptop. The application directly reads and writes to the database file.

**Diagram:**

```
[User/Application] <----> [Database]
```

**Code Example (Python + SQLite):**

```python
import sqlite3
conn = sqlite3.connect('notes.db')
cursor = conn.cursor()
cursor.execute("CREATE TABLE IF NOT EXISTS notes (id INTEGER, text TEXT)")
cursor.execute("INSERT INTO notes VALUES (1, 'DBMS Architecture')")
conn.commit()
conn.close()
```

- No intermediate layer; the app directly manipulates the database.

---

## 2-Level Architecture

- **Two-tier architecture**
- Consists of:
  - **External Level:** User views
  - **Internal Level:** Physical storage
- Provides some data abstraction.
- Users interact with the database through application programs.

### Deep Example

**Scenario:**  
A company uses a desktop application to manage employee records. The app provides forms and reports (external level), while the database stores the actual data (internal level).

**Diagram:**

```
[User/Application] <----> [DBMS Engine] <----> [Database Storage]
```

**Code Example (Java + JDBC):**

```java
// External Level: User interacts with GUI
// Internal Level: JDBC connects to MySQL database
Connection conn = DriverManager.getConnection("jdbc:mysql://localhost:3306/company", "user", "pass");
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM employees");
while(rs.next()) {
    System.out.println(rs.getString("name"));
}
conn.close();
```

- The user never sees the physical storage details; the application abstracts it.

## 3-Level Architecture

- **Three-tier architecture**
- Consists of:
  - **External Level:** User views
  - **Conceptual Level:** Logical structure of the entire database
  - **Internal Level:** Physical storage
- Highest level of data abstraction.
- Separates user views, logical schema, and physical storage.

### Deep Example

**Scenario:**  
A university uses a web-based student management system.

- Students see their own grades (external level).
- The system maintains a logical schema for courses, grades, and students (conceptual level).
- Data is stored in physical files or disks (internal level).

**Diagram:**

```
[User Interface] <----> [Application Server] <----> [DBMS Engine] <----> [Database Storage]
      |                    | (Conceptual Level)         | (Internal Level)
(External Level)
```

**Code Example (Web App):**

- **External Level:** HTML/CSS/JS for user interface
- **Conceptual Level:** ORM (Object Relational Mapping) defines models for students, courses, grades
- **Internal Level:** DBMS stores tables and indexes on disk

**Example (Django ORM):**

```python
# Conceptual Level: Models
class Student(models.Model):
    name = models.CharField(max_length=100)
    grade = models.IntegerField()
# External Level: Views/Forms
# Internal Level: Data stored in PostgreSQL
```

- The user interacts with forms, the system uses models, and the DBMS stores data physically.

## Comparison Table

| Architecture | Levels                             | Abstraction | Use Case                   |
| ------------ | ---------------------------------- | ----------- | -------------------------- |
| 1-Level      | 1 (User-DB)                        | None        | Simple, small applications |
| 2-Level      | 2 (External, Internal)             | Partial     | Medium complexity          |
| 3-Level      | 3 (External, Conceptual, Internal) | Full        | Large, complex systems     |
