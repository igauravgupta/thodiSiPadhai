The Entity-Relationship Model (ER Model) is a conceptual model for designing a databases. This model represents the logical structure of a database, including entities, their attributes and relationships between them.

## Key Concepts

- **Entity:** An objects that is stored as data such as Student, Course or Company.
- **Attribute:** Properties that describes an entity such as StudentID, CourseName, or EmployeeEmail.
- **Relationship:** A connection between entities such as "a Student enrolls in a Course".

## What is an Entity?

An Entity represents a real-world object, concept or thing about which data is stored in a database. It act as a building block of a database. Tables in relational database represent these entities.

---

## Types of Entity

There are two main types of entities:

### 1. Strong Entity

A Strong Entity is a type of entity that has a key Attribute that can uniquely identify each instance of the entity. A Strong Entity does not depend on any other Entity in the Schema for its identification. It has a primary key that ensures its uniqueness and is represented by a rectangle in an ER diagram.

### 2. Weak Entity

A Weak Entity cannot be uniquely identified by its own attributes alone. It depends on a strong entity to be identified. A weak entity is associated with an identifying entity (strong entity), which helps in its identification. A weak entity are represented by a double rectangle. The participation of weak entity types is always total. The relationship between the weak entity type and its identifying strong entity type is called identifying relationship and it is represented by a double diamond.

#### Example

A company may store the information of dependents (Parents, Children, Spouse) of an Employee. But the dependents can't exist without the employee. So dependent will be a Weak Entity Type and Employee will be identifying entity type for dependent, which means it is Strong Entity Type.

## Relationship Set

A Relationship Set is a collection of similar relationships in an ER Model. Each relationship in the set connects entities from entity sets according to a specific association. For example, the "Enrolled" relationship set might connect students and courses, representing which students are enrolled in which courses. Relationship sets are depicted by diamonds in ER diagrams, and they can have attributes of their own, such as the date of enrollment.

---

## Degree of Relationship

The Degree of a Relationship refers to the number of entity sets that participate in a relationship set. The most common degrees are:

- **Unary (Degree 1):** The relationship involves only one entity set. For example, an "manages" relationship where an employee manages other employees.
- **Binary (Degree 2):** The relationship involves two entity sets. For example, "enrolls" between Student and Course.
- **Ternary (Degree 3):** The relationship involves three entity sets. For example, a "supplies" relationship among Supplier, Product, and Warehouse.

Higher-degree relationships (more than three) are rare in practice.

## Cardinality in ER Model

Cardinality defines the number of instances of one entity that can be associated with the number of instances of another entity via a relationship. Cardinality constraints help specify the nature of relationships:

- **One-to-One (1:1):** Each entity in set A is related to at most one entity in set B, and vice versa.
- **One-to-Many (1:N):** Each entity in set A can be related to multiple entities in set B, but each entity in set B is related to at most one entity in set A.
- **Many-to-One (N:1):** Each entity in set B can be related to multiple entities in set A, but each entity in set A is related to at most one entity in set B.
- **Many-to-Many (M:N):** Entities in set A and set B can be related to multiple entities in each other.

Cardinality is crucial for designing the structure and integrity of a database, as it determines how data is linked and retrieved.

## Recursive Relationships in ER diagrams

A **Recursive Relationship** in an ER diagram occurs when an entity set participates more than once in a relationship set, but in different roles. This means an entity is related to other entities of the same entity set.

### Example

Consider an Employee entity set with a "manages" relationship. An employee can manage other employees, so both the manager and the subordinate are from the same entity set.

- **Entity:** Employee
- **Relationship:** Manages (an employee manages another employee)

In ER diagrams, recursive relationships are represented by connecting the entity set to itself through a diamond (relationship) and labeling the roles (e.g., "manager" and "subordinate").

### Use Cases

- Employees supervising other employees
- Courses having prerequisites (a course requires another course)
- Parts assembled from other parts

Recursive relationships help model hierarchical or network structures within a single entity set.
