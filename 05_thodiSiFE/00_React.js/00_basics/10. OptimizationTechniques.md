#### Code-splitting and Lazy loading

- What it is:

  - Load resources (components, images, data) only when needed instead of upfront.
  - Breaks the app into smaller bundles (chunks) so initial bundle is smaller and loads faster.

- Benefits:

  - Faster initial load and better perceived performance.
  - Reduced memory/CPU for rarely-used code.
  - Easier parallel network fetches for large apps.

- How it works (high level):

  - Webpack/Rollup/Vite turn dynamic `import()` into separate chunks.
  - The chunk is fetched when the code path requiring it runs.
  - Use route-level splitting for large pages and component-level lazy loading for rarely used UI.

- React.lazy + Suspense (basic):

  - React.lazy accepts a dynamic import that must resolve to a default export.
  - Suspense shows a fallback while the lazy component is loading.
  - Use an Error Boundary to handle loading failures.

  ```jsx
  import React, { Suspense, lazy } from "react";

  const About = lazy(() => import("./About")); // default export required

  function App() {
    return (
      <div>
        <h1>Welcome to My App</h1>
        <Suspense fallback={<h2>Loading...</h2>}>
          <About />
        </Suspense>
      </div>
    );
  }

  export default App;
  ```

- Route example (React Router v6):

  ```jsx
  const Home = lazy(() => import("./Home"));
  const About = lazy(() => import("./About"));

  // inside routes
  <Routes>
    <Route path="/" element={<Home />} />
    <Route
      path="/about"
      element={
        <Suspense fallback={<Spinner />}>
          <About />
        </Suspense>
      }
    />
  </Routes>;
  ```

- Best practices:

  - Prefer route-level splitting for large pages; component-level for rarely used widgets.
  - Add Error Boundaries around lazy components.
  - Avoid over-splitting — measure with Lighthouse/React DevTools.
  - Consider prefetching/pReloading for likely-to-be-used routes.
  - For SSR, use libraries like @loadable/component or framework-specific tools.

- Notes:
  - React.lazy works only for default exports. For named exports wrap or re-export as default.
  - Suspense for data fetching requires concurrent features or libraries that support Suspense.

### Avoid unnecessary re-renders

- Techniques:
  - Memoize components: React.memo(Component).
  - Stabilize props: useCallback for handlers, useMemo for objects/arrays passed as props.
  - Immutable updates so shallow comparisons detect changes.
  - Use keys correctly to avoid remounts.
  - For class components: shouldComponentUpdate / PureComponent.
  - Profile with React DevTools to find real hotspots.

### Single Responsibility Principle (SRP)

- Keep components focused: one component = one responsibility (presentation, a container, or a small composite).
- Move non-UI logic into custom hooks (e.g., useAuth, useFetch). This:
  - Improves testability and reuse.
  - Makes components simpler and easier to reason about.
- If not folloed - hard to test components

### What are Higher-Order Components (HOCs)?

- Definition: a function that takes a component and returns a new component with added behavior.
  - Signature example: const withFeature = WrappedComponent => props => <WrappedComponent {...props} extraProp={...} />;
- Use cases: cross-cutting concerns like authentication, theming, data fetching, analytics.
- Notes:
  - HOCs wrap and enhance behavior without modifying the original component.
  - Preserve displayName for better DevTools output.
  - Alternatives: hooks and render props — hooks are usually preferred for new code because they’re simpler to compose.
- Example:
  - const withAuth = Wrapped => props => { const user = useAuth(); return user ? <Wrapped {...props} /> : <Login />; };
