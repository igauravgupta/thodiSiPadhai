# useMemo()

## What it is
- `useMemo` is a React hook that memoizes a computed value.
- Signature: `const memoized = useMemo(() => compute(), [deps])`.
- It returns the cached result of the function until a dependency changes.
- It is a performance optimization (a hint), not a guarantee; it runs during render and must be pure (no side effects).

## When to use (use cases)
- Expensive computations that run on every render (heavy calculation, filtering/sorting large lists).
- Avoiding re-creating stable references (objects, arrays, functions) so that child components relying on referential equality do not re-render.
- Combined with `React.memo` to prevent unnecessary child re-renders.

## When NOT to use
- For cheap calculations — premature optimization adds complexity.
- Not a replacement for proper memoization libraries where appropriate.
- Don’t rely on it for side effects — use `useEffect` for side effects.

## Important notes
- Dependencies must include all values used inside the compute function.
- Referential equality matters: objects/arrays/functions are compared by reference.
- `useMemo` may re-run on every render in development (strict mode) — treat it as a performance hint.

## Examples

1) Expensive calculation
```jsx
import React, { useMemo, useState } from 'react';

function expensiveFib(n) {
    // naive slow Fibonacci for demonstration
    if (n < 2) return n;
    return expensiveFib(n - 1) + expensiveFib(n - 2);
}

export default function FibDemo() {
    const [count, setCount] = useState(35);
    const [ticks, setTicks] = useState(0);

    const fibValue = useMemo(() => expensiveFib(count), [count]);

    return (
        <div>
            <div>Fib({count}) = {fibValue}</div>
            <button onClick={() => setTicks(t => t + 1)}>Re-render ({ticks})</button>
            <button onClick={() => setCount(c => c + 1)}>Increase n</button>
        </div>
    );
}
```
Without `useMemo`, clicking “Re-render” would recompute the expensive function each time.

2) Stabilizing object/array passed to memoized child
```jsx
import React, { useMemo, useState } from 'react';
const Child = React.memo(({ config }) => {
    console.log('Child render');
    return <div>{config.label}</div>;
});

export default function Parent() {
    const [count, setCount] = useState(0);
    const config = useMemo(() => ({ label: 'Hello' }), []);
    return (
        <div>
            <Child config={config} />
            <button onClick={() => setCount(c => c + 1)}>Re-render parent ({count})</button>
        </div>
    );
}
```
If `config` were recreated on each render, `Child` would re-render even though its visible data didn't change.

## Senerio 
When a page supports a theme toggle (light/dark) and also performs a heavy calculation during render, toggling the theme can cause a full re-render and re-run the expensive computation even when its inputs haven’t changed. This is common when the heavy work is done directly in the component body or in a function that runs on every render.

To avoid repeated work:
- Move the expensive computation into useMemo and include only the values that actually affect the result in its dependency array (not the theme).
- If the result never changes after initial load, compute it once (lazy initialize state or useMemo with an empty dependency array).
- For very heavy CPU work, consider offloading to a web worker or performing the calculation outside the render path and storing/cacheing the result (module-level cache, IndexedDB, or a memoization library).
- Use React.memo for child components that only depend on the computed value so theme changes don’t force them to re-render unnecessarily.
- Always measure (profiling) before and after to confirm the optimization helps, and be aware useMemo is a hint — in Strict Mode it may run more frequently during development.

Example pattern:
```jsx
// heavyInput is the only value that should trigger recompute
const heavyResult = useMemo(() => computeHeavy(heavyInput), [heavyInput]);
```
This ensures toggling theme (or other unrelated state) will not recompute the heavy work.

## Quick checklist
- Use for expensive derived values or to keep stable references.
- Always include dependencies.
- Keep the compute function pure and side-effect free.
- Prefer readability; only optimize when you have measurable cost.
