# Props

Props (properties) are the inputs you pass from a parent component to a child component in React. Think of them as function arguments for components. Important points:

- Props are read-only inside the child — components should not mutate their props.
- You can pass primitive values, objects, arrays, functions, JSX, and even other components as props.
- Access props via the `props` object (or `this.props` in class components) or by destructuring in the parameter list.
- Use the spread operator (`...`) to forward many props at once.

## Basic example (functional component)

Passing props from parent to child:

```jsx
// Parent.jsx
import React from "react";
import Greeting from "./Greeting";

function Parent() {
  return <Greeting name="Gaurav" age={30} />;
}

export default Parent;
```

Receiving props and using them (destructuring in the function signature):

```jsx
// Greeting.jsx
import React from "react";

function Greeting({ name, age }) {
  return (
    <div>
      <p>Hello, {name}!</p>
      <p>Age: {age}</p>
    </div>
  );
}

export default Greeting;
```

You can also access props without destructuring:

```jsx
function Greeting(props) {
  return <p>Hello, {props.name}!</p>;
}
```

## Passing many props with spread (convenient shorthand)

```jsx
const person = { name: "Gaurav", age: 30, city: "Pune" };
<Greeting {...person} />;
```

This is equivalent to `<Greeting name="Gaurav" age={30} city="Pune" />`.

## Default props and prop validation (optional)

Set defaults in modern functional components:

```jsx
function Greeting({ name = "Guest", age = "Unknown" }) { ... }
```

For runtime prop validation you can use the `prop-types` package:

```jsx
import PropTypes from "prop-types";

Greeting.propTypes = {
  name: PropTypes.string.isRequired,
  age: PropTypes.number,
};
```

## Class component example

```jsx
class Greeting extends React.Component {
  render() {
    // access via this.props
    const { name, age } = this.props;
    return (
      <p>
        {name} — {age}
      </p>
    );
  }
}
```

Summary:

- Use props to pass data and callbacks down the component tree.
- Prefer destructuring for clarity.
- Keep props immutable inside the child; use state for internal mutable data.
- Use spread to forward many props, and default values or prop-types for robustness.

## Props with key

When rendering many instances of the same child (e.g., mapping an array to components) you should provide a key for each item. Keys help React identify which items changed, were added, or were removed so it can efficiently update the DOM (reconciliation).

Why keys matter

- Keys are used by React internally during reconciliation to match elements between renders. Stable keys reduce unnecessary DOM updates and preserve component state.
- Keys must be unique among siblings and stable across renders (don’t change between renders for the same item).

Important notes

- The special key prop is not available inside the child via props; it’s only for React. If the child needs the id, pass it as a separate prop (e.g., id={item.id}).
- Avoid using array index as key when the list can be reordered, filtered, or items inserted/removed. Index keys can cause incorrect reuse of components and UI bugs.
- For moderate lists (15–20 items) keys are sufficient for good performance. If you have many hundreds or thousands, consider virtualization (react-window/react-virtualized) or pagination.
- To avoid unnecessary re-renders for expensive children, combine stable keys with React.memo (functional) or PureComponent (class).

Good example

```jsx
// Parent.jsx
function Parent({ items }) {
  return (
    <ul>
      {items.map((item) => (
        // key for React; pass id separately if child needs it
        <Item key={item.id} item={item} id={item.id} />
      ))}
    </ul>
  );
}

// Item.jsx
function Item({ item }) {
  return <li>{item.title}</li>;
}
```

Bad example (index as key — risky if list changes)

```jsx
{
  items.map((item, index) => (
    <Item key={index} item={item} /> // bad when items reorder or are inserted/removed
  ));
}
```

When you have 15–20 identical children

- This is a small list; keys + normal React rendering is usually fine.
- If each child is expensive to render, wrap the child in React.memo or use useCallback for handlers passed as props to avoid unnecessary work.
- If the UI shows many off-screen items or you expect growth, add list virtualization.

Summary

- Always give stable, unique keys for list items.
- Don’t rely on key to pass data into the child — pass needed values as props.
- Use memoization and virtualization when rendering becomes a performance problem.

## Problems with Props

- Common problems with props

  - Prop drilling: passing props through many intermediate components that don't use them, making code harder to maintain.
  - Tight coupling: components become interdependent on specific prop shapes, reducing reusability.

- How to handle props effectively (basics, theory only)
  - Lift state appropriately: place state at the nearest common ancestor that logically owns the data.
  - Reduce prop drilling: move shared data to a higher-level provider or component so intermediates don’t need to forward values.
  - Adopt a single-source-of-truth approach: centralize important application state to avoid duplicated and conflicting copies.
