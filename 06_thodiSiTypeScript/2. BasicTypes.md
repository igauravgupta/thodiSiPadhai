## Tuples

A tuple is a fixed-length array where each element has a specific type and position. Use tuples when you want a known number of elements with different types.

Example — basic tuple:
```ts
const person: [string, number] = ["Alice", 30];

console.log(person[0].toUpperCase()); // "ALICE"
console.log(person[1].toFixed(0));    // "30"

// destructuring
const [name, age] = person;
```

Notes and variants:
- Optional elements: `[string, number?]` allows the second item to be absent.
- Rest elements: `type Row = [string, ...number[]]` — first is string, rest are numbers.
- Readonly tuple: `const point: readonly [number, number] = [0, 0];` prevents mutation.
- Be aware: some array methods (like push) can still be called on tuples at runtime, which may bypass compile-time length checks.

Tuples are useful for representing records with a fixed shape (e.g., [label, value, timestamp]) while keeping element types strict.

## Enums

Enums provide a way to define a set of named constants. They can be numeric or string-based and are useful when you want readable names for a fixed set of values.

Basic numeric enum (has reverse mapping at runtime):
```ts
enum Direction {
    Up,    // 0
    Down,  // 1
    Left,  // 2
    Right  // 3
}

console.log(Direction.Up);    // 0
console.log(Direction[0]);    // "Up"
```

Custom numeric values:
```ts
enum Status {
    Success = 1,
    Failure = 0,
    Pending = 2
}
```

String enums (no reverse mapping):
```ts
enum Color {
    Red = "RED",
    Green = "GREEN",
    Blue = "BLUE"
}

console.log(Color.Red); // "RED"
```

Const enums (inlined and removed from emitted JS — good for performance):
```ts
const enum LogLevel {
    Debug,
    Info,
    Warn,
    Error
}

const level = LogLevel.Warn; // compiled directly to the numeric value
```

Notes and best practices:
- Numeric enums produce a runtime object and support reverse lookup (Direction[0] -> "Up"). String enums do not.
- Heterogeneous enums (mixing string and number) are allowed but generally discouraged.
- For many cases, union string literal types (type Dir = "Up" | "Down") can be simpler and produce cleaner output.
- Use const enums when you want values inlined and no runtime enum object.
- Enums are useful for modeling a small, known set of related values with clear names.


## Special types: any, unknown, void, null, undefined, never

These are TypeScript’s special-purpose types used for cases that don’t fit the normal primitive/reference types.

### any
- Means “opt out” of type checking — any operation is allowed.
- Useful for gradual migration or interoperating with untyped libraries.
- Avoid when possible because it forfeits compiler safety.

Example:
```ts
let x: any = JSON.parse(someString);
x.doSomething(); // no compile error
```

### unknown
- Safer alternative to `any`. You can assign anything to `unknown`, but you must narrow it before using.
- Forces explicit type checks/casts, improving safety.

Example:
```ts
let value: unknown = getInput();
if (typeof value === "string") {
    console.log(value.toUpperCase());
}
```

### void
- Used as the return type for functions that do not return a value.
- In practice, `void` is similar to `undefined` for return values but is intended for functions.

Example:
```ts
function log(msg: string): void {
    console.log(msg);
}
```

### null and undefined
- Represent absence of a value.
- Under `strictNullChecks` they are distinct types and are not assignable to other types unless explicitly allowed (e.g., `string | null`).
- Use unions to model optional values.

Examples:
```ts
let a: null = null;
let b: undefined = undefined;

let maybe: string | null = Math.random() > 0.5 ? "ok" : null;
```

Notes:
- `undefined` typically means “not initialized”; `null` typically means “explicitly no value”.
- Prefer `T | null` or `T | undefined` rather than allowing them implicitly.

### never
- Represents values that never occur — e.g., functions that always throw or infinite loops.
- Useful for exhaustive checks in discriminated unions: a variable of type `never` indicates an impossible branch.

Examples:
```ts
function fail(msg: string): never {
    throw new Error(msg);
}

function exhaustiveCheck(x: never): never {
    return x;
}
```

Best practices
- Prefer `unknown` over `any` when you need to accept arbitrary values but still want type safety.
