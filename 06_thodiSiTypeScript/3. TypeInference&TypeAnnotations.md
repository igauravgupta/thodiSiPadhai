# Type Inference and Type Annotations

## Type Inference

Type inference: TypeScript figures out a value’s type from the value or how it’s used, so you often don’t need to write the type.

### Basic examples

#### Variable inference

```ts
let x = 10; // inferred as number
const y = "hello"; // inferred as "hello" (literal type for const)
let z; // inferred as any (or unknown depending on compiler settings)
```

#### Function return inference

```ts
function add(a: number, b: number) {
  return a + b; // return type inferred as number
}
```

#### Parameter inference from context

```ts
const nums: number[] = [1, 2, 3];
nums.map((n) => n * 2); // n inferred as number from array element type
```

---

### When TypeScript figures out types

- From values: literals like numbers, strings, booleans.
- From what a function returns.
- From context: callbacks, assignments, and other surrounding types.
- For arrays/mixed values it picks a "best common" type.
- From generics: inferred from the call-site when possible.

---

## Type Annotations

Type annotations: You write types yourself to make intent clear or to restrict what values are allowed (e.g., function params, return types, or variables).

### Basic examples

#### Variable's

```ts
// Good: annotate public API
export function parseConfig(input: string): Config { ... }

// Good: annotate external data
const payload: unknown = JSON.parse(raw);
```

```ts
let count: number = 0;
const name: string = "Gaurav";
let maybe: boolean | null = null;
```

### Function parameters and return type

```ts
function greet(name: string): string {
  return `Hello, ${name}`;
}

const multiply = (a: number, b: number): number => a * b;
```

#### Objects, interfaces and type aliases

```ts
interface User {
  id: number;
  name: string;
  email?: string; // optional
  readonly createdAt: Date;
}

type ID = number | string;

const user: User = { id: 1, name: "Alice", createdAt: new Date() };
```

#### Arrays and tuples

```ts
const nums: number[] = [1, 2, 3];
const pair: [string, number] = ["age", 30];
```

#### Union and intersection

```ts
type Success = { ok: true; data: string };
type Failure = { ok: false; error: string };
type Result = Success | Failure;

type WithMeta = User & { meta: Record<string, unknown> };
```

#### Generics

```ts
function identity<T>(value: T): T {
  return value;
}

const boxed: string = identity<string>("hello");
```

#### Callbacks and function types

```ts
type Comparator = (a: number, b: number) => number;
function sort(nums: number[], cmp: Comparator) {
  /* ... */
}
```

#### unknown, any and type assertions

```ts
const raw: unknown = JSON.parse(someJson);
if (typeof raw === "object" && raw !== null) {
  const obj = raw as Record<string, unknown>;
  // further validation...
}

let flexible: any = 42;
flexible = "now a string"; // any bypasses checks
```

#### Function overloads (API-like examples)

```ts
function parse(input: string): number;
function parse(input: string, asFloat: true): number;
function parse(input: string, asFloat?: boolean) {
  return asFloat ? parseFloat(input) : parseInt(input, 10);
}
```

#### Practical: annotate an API response

```ts
type Post = { id: number; title: string; body: string };
async function fetchPosts(url: string): Promise<Post[]> {
  const res = await fetch(url);
  const data: unknown = await res.json();
  // validate/transform data here
  return data as Post[]; // prefer proper runtime validation in real code
}
```

### When to add type annotations

- For public/exported functions and module boundaries.
- When return types are complex (unions, specific tuples).
- For external data (API responses) — mark as unknown or a specific type and validate.
- When you need a stricter type than inference gives.
- To make your intent clearer for future readers.
